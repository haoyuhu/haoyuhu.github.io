## 问题界定

当内置整数类型会溢出时，更实际的做法是把数字当作十进制字符串，并手工实现算术运算。原文梳理了三类算法。这个结构今天仍然成立，但中间那一类值得一个更准确的名字：它本质上就是 **Karatsuba 风格乘法**。

| 方法 | 核心思路 | 时间复杂度 | 最佳适用场景 |
| --- | --- | --- | --- |
| 竖式乘法 | 模拟纸笔乘法 | `O(n^2)` | 面试、短小且可靠的实现 |
| Karatsuba 风格分治 | 将 4 次子乘法降为 3 次 | `O(n^{log_2 3}) ≈ O(n^1.585)` | 学习算法优化 |
| FFT / NTT | 将乘法视为多项式卷积 | `O(n log n)` | 超大输入、专用库 |

## Karatsuba 风格拆分

设

`P = A * 10^k + B`  
`Q = C * 10^k + D`

其中 `k` 是低半部分的位数。如果两者长度不同，一个实用的选择是 `k = min(len(P), len(Q)) / 2`。

直接按分治展开可得：

`PQ = AC * 10^(2k) + (AD + BC) * 10^k + BD`

但这仍然需要 **四次** 递归乘法：`AC`、`AD`、`BC` 和 `BD`，因此渐近复杂度依旧是 `O(n^2)`。

真正有用的恒等式是：

`(A - B)(C - D) = AC - AD - BC + BD`

于是中间项可以改写为：

`AD + BC = AC + BD - (A - B)(C - D)`

从而得到：

`PQ = AC * 10^(2k) + (AC + BD - (A - B)(C - D)) * 10^k + BD`

现在每一层只需进行 **三次** 递归乘法：`AC`、`BD` 和 `(A-B)(C-D)`。递推式变为：

`T(n) = 3T(n/2) + O(n)`

根据主定理，这就是 **`O(n^{log_2 3}) ≈ O(n^1.585)`**，而不是 `O(n^1.5)`。

## 决定实现成败的辅助例程

乘法公式很简洁；真正的工程量都在辅助函数上。

- **`addStrings`**：从右向左扫描，维护进位，逆序追加数字，最后统一反转一次。
- **`subtractStrings`**：先比较绝对值，必要时交换两数，再逐位借位相减。返回绝对值结果和符号。
- **`multiplyByDigit`**：当某个操作数长度为 `1` 时，这是最简单的基本情形。
- **`trimLeadingZeros`**：在递归中切分子串后必不可少；否则像 `"00012"` 这样的值会持续渗入后续步骤。
- **`shift10(s, k)`**：通过在末尾补上 `k` 个零来表示乘以 `10^k`，而不是去构造巨大的数值幂。

有几个小改动，可以让原始代码更干净也更安全：

1. 把像 `minus` 这样含糊的辅助函数名改成 `subtractStrings`，并修正 `mulitplySingleNumber` 之类的拼写错误。
2. 显式处理两个数相等时的减法：如果绝对值相同，立即返回 `"0"`。
3. 在合并 `(A-B)` 和 `(C-D)` 的符号时，用 `sameSign = (sign1 == sign2)` 代替冗长的布尔表达式。
4. 每次拆分后以及最终重组后都要修剪前导零。
5. 如果更关心性能，最好把数字存成逆序的 `vector<int>`，而不是在整个递归过程中反复在 `char` 和 `int` 之间转换。

## 实践结论

对于真正的 **LeetCode `Multiply Strings`** 题目，竖式 `O(n^2)` 解法通常是最好的权衡：代码更短，边界情况更少，而且在面试规模的输入上性能已经很强。但如果目标是理解乘法复杂度如何随规模变化，Karatsuba 风格的改写就是顺理成章的下一步。它保持了同样的十进制字符串模型，同时也说明：**大部分复杂性其实都在表示方式、归一化和辅助操作上——而不在最后那条代数恒等式本身**。
